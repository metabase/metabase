<objective>
Create the query-log around-middleware that pushes correlation metadata into Log4j2 ThreadContext and emits an always-on INFO summary log line at query completion. Wire it into the QP pipeline. Write tests covering summary content, ThreadContext propagation, and concurrent request isolation.

Purpose: This is the foundation for all structured query logging. The correlation ID propagation via ThreadContext enables all subsequent per-statement logging to be correlated. The summary line alone delivers immediate value -- admins can see who ran what, where, and how long it took.

Output: New namespace `metabase.query-processor.middleware.query-log`, updated QP `around-middleware` vector, comprehensive test suite.
</objective>

Create `src/metabase/query_processor/middleware/query_log.clj`:

1. Namespace declaration requiring: `metabase.util.log`, `metabase.config.core`, `metabase.query-processor.schema`, `metabase.query-processor.util`, `metabase.util.malli`.

2. Define a private helper `query-log-context` that extracts the ThreadContext map from a query:
   - `:request-id` from `config/*request-id*` (or `(str (random-uuid))` if nil)
   - `:database-id` from `(:database query)`
   - `:card-id` from `(get-in query [:info :card-id])` (only if non-nil)
   - `:dashboard-id` from `(get-in query [:info :dashboard-id])` (only if non-nil)
   - `:user-id` from `(get-in query [:info :executed-by])`
   - `:context` from `(get-in query [:info :context])`
   Return a map with only non-nil values (use `cond->` or filter nils).

3. Define a private helper `emit-summary!` that takes the context map, start-time-ms (long), and row-count (volatile long). Emits a single `log/info` line with format:
   `"Query completed :: request-id=%s database=%s user=%s card=%s dashboard=%s context=%s queries=1 rows=%d time=%dms"`
   Omit card/dashboard fields if nil. Use simple `str` concatenation (not `format`) for performance per Pitfall 4. The function should be pure string building -- no expensive computation.

4. Define the public `query-log-middleware` function with the around-middleware signature `(fn [qp] -> (fn [query rff] -> result))`:
   - Check `(qp.util/userland-query? query)` -- if not userland, pass through to `(qp query rff)`
   - Extract context via `query-log-context`
   - Record `start-time-ms` via `(System/currentTimeMillis)`
   - Create a `row-count` volatile (vswap! to increment)
   - Wrap the inner call with `(log/with-thread-context context-map ...)`
   - Create a wrapping rff that delegates to the original rff but wraps the returned rf:
     - In the completing arity `([acc])`: call `emit-summary!` with context, start-time, row-count, then delegate to inner rf completing arity
     - In the reducing arity `([result row])`: increment row-count via vswap!, delegate to inner rf
   - Call `(qp query wrapped-rff)`

Key design decisions:
- Use `log/with-thread-context` which auto-prefixes keys with "mb-" and handles cleanup in finally (addresses Pitfall 3: MDC leaking)
- Emit summary synchronously in completing arity, NOT in async save path (addresses Pitfall architecture anti-pattern 3)
- Generate fallback UUID when `*request-id*` is nil for non-HTTP contexts (addresses research gap)
- Keep `queries=1` for now -- Phase 2 may need to track multi-statement counts via an atom shared with the JDBC layer
- Use `log/info` which checks log level before evaluating args (addresses Pitfall 4: performance)
