Purpose: This is the core "see what queries Metabase sends" feature. Combined with Phase 1's ThreadContext correlation, users can grep by request-id to see both the summary and all individual SQL statements for a request.

Output: Modified execute.clj with DEBUG logging, plus unit tests

**Practical approach:** Create a new private function `emit-statement-detail!` that takes sql, params, database-id, and elapsed-ms and emits the DEBUG log line. Test this function directly (it's a pure side-effect function). Then wire it into `execute-reducible-query`.

In `execute.clj`, modify `execute-reducible-query` (around line 789-790):

1. Before calling `execute-statement-or-prepared-statement!`, capture a timer with `(u/start-timer)`
2. After the call returns the ResultSet, compute elapsed time with `(u/since-ms timer)`
3. Emit a DEBUG log line using `log/debug` with StringBuilder (matching Phase 1 pattern):
   ```
   Statement executed :: database=N SQL: {full-sql-text} params=[v1, v2] time=Nms
   ```
4. Use `(log/is-loggable? :debug)` or just rely on `log/debug` being a no-op at INFO level (which it is -- Log4j2 short-circuits). For the StringBuilder construction cost, wrap the entire block in `(when (log/is-debug?) ...)` to avoid string construction when DEBUG is disabled.

**Important implementation details:**
- Use `u/start-timer` and `u/since-ms` (NOT `System/currentTimeMillis`) per Phase 1 linter finding
- Use `StringBuilder` for log line construction per Phase 1 pattern
- The database ID is available as `(:id database)` where `database` is already bound on line 776
- Omit params from the log line when params is empty/nil (cleaner output)
- Do NOT truncate SQL -- users explicitly opted in to DEBUG logging, they want the full query
- Do NOT truncate params -- same reasoning, though note PII risk (documented in research as acceptable for DEBUG-only)
- The ThreadContext already contains request-id from Phase 1 middleware, so it will appear in log output automatically via the `%notEmpty{%X}` pattern -- do NOT redundantly include request-id in the log message itself

**Log format:**
```
Statement executed :: database=2 SQL: SELECT "PUBLIC"."ORDERS"."ID" ... LIMIT 2000 params=[1, "active"] time=230ms
```
When no params:
```
Statement executed :: database=2 SQL: SELECT "PUBLIC"."ORDERS"."ID" ... LIMIT 2000 time=230ms
```
